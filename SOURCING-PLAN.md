## 3. Scoring Framework

Each target is scored 1-5 on each dimension, then ranked by composite score:

| Dimension | Weight | What we measure |
|-----------|--------|-----------------|
| Sector thesis fit | 20% | Does the company sit in a sector with clear tailwinds? |
| Revenue quality | 20% | Recurring revenue % / gross margin / churn |
| Growth trajectory | 15% | YoY revenue growth / user growth / market share gains |
| Team quality | 15% | Founder background, domain expertise, exec completeness |
| Defensibility | 10% | IP, network effects, switching costs, brand, data moat |
| Capital efficiency | 10% | Burn multiple / cash runway / path to profitability |
| Exit optionality | 5% | Strategic buyer landscape / M&A market depth |
| Risk | 5% | Customer concentration / regulatory / key-person / tech debt |

**Thresholds:**
- Score ≥ 4.0: Active pursuit — brief and present to Managing Partner
- Score 3.0–3.9: Watchlist — monitor quarterly
- Score < 3.0: Cold — archive unless re-triggered

## 4. Pipeline Stages

| Stage | Definition | Action |
|-------|------------|--------|
| Sourced | Company identified in scan | Log name, sector, source, one-line thesis |
| Screened | Quick pass (public data) — passes grade? | Research score, assign preliminary score |
| Researched | Deep dive completed | Write 1-page brief, assign final score |
| Watchlist | Score 3.0–3.9 or incomplete data | Set quarterly review reminder |
| Active | Score ≥ 4.0, ready for partner review | Brief Managing Partner |
| Outreach | Partner approved initial contact | Delegate to partner; analyst role pauses |
| Dead | No longer meets criteria | Document reason, archive |

## 7. Pipeline Tracking in DealDesk

DealDesk has two complementary tracking layers for the sourcing pipeline:

### 7.1 Pipeline Stage Labels (Issue Labels)

Six pipeline stage labels exist on the 123 Capital company in DealDesk. These are applied to issues that track individual targets:

| Stage | Label | Color | Hex | Definition |
|-------|-------|-------|-----|------------|
| 1 | Identified | Blue | `#3B82F6` | Target spotted in scan — logged with one-line thesis |
| 2 | Contacted | Amber | `#F59E0B` | Initial outreach sent by sector sourcer |
| 3 | Meeting | Purple | `#8B5CF6` | Introductory call scheduled or completed |
| 4 | LOI | Teal | `#14B8A6` | Letter of Intent issued or received |
| 5 | Diligence | Indigo | `#6366F1` | Formal due diligence underway |
| 6 | Closed | Green | `#10B981` | Deal completed or abandoned |

All 6 labels are created and operational. Use `GET /companies/:companyId/labels` to verify.

### 7.2 Structured Deal Pipeline (ddTargets)

The primary deal tracking system uses the `dd_targets` table with a dedicated status enum:

| Status | Definition |
|--------|------------|
| `sourced` | Target logged via createTarget tool |
| `qualified` | Fit score validated, passes threshold |
| `contacted` | Initial outreach sent |
| `replied` | Prospect responded |
| `meeting_booked` | Call or meeting scheduled |
| `in_diligence` | Formal due diligence underway |
| `passed` | Evaluated but not pursued further |
| `closed_won` | Deal completed |
| `closed_lost` | Deal lost to competitor or abandoned |

Targets are added via the **createTarget** tool (API or agent tool) with structured fields:
companyName, website, sector, subSector, hqCity, hqState, estimatedRevenue, fitScore (0-100, minimum 40), fitRationale, and sources with URLs.

The Pipeline page (`/deal-desk/pipeline`) is the primary kanban tracker by stage. The Targets page (`/deal-desk/targets`) displays the same records as a sortable table and supports status transitions via the detail sheet. The "Source targets now" button creates a sourcing ticket assigned to the Sector Sourcer agent.

### 7.3 Adding a New Target

**Via the createTarget tool (primary — for sector sourcers/agents):**
```
POST /:companyId/deal-desk/targets
{
  thesisId: "<uuid>",
  companyName: "<name>",
  website: "<url>",
  sector: "<sector>",
  fitScore: <40-100>,
  fitRationale: "<why this fits>",
  sources: [{ url: "<source>", description: "<how found>" }]
}
```

**Via issue (secondary — for manual tracking):**
1. Create an issue with title `[Target] <Company Name> — <Sector>`
2. Apply the **Identified** label
3. Fill the description with relevant context (sector, source, thesis)

### 7.4 Moving Through Pipeline

- Update the **label** on the issue when a target advances (Identified → Contacted → etc.)
- Update the **ddTargets status** via the Targets UI (`/deal-desk/targets`) or PATCH API when the deal stage changes
- Add **comments** on the issue for key events: outreach sent, meeting notes, LOI issued
- Change issue **status** to `done` when a deal closes or is abandoned
- Add a note explaining why if archived

### 7.5 Weekly Review

Each Friday, run a pipeline health check:
1. List all targets via the Targets dashboard (or `listTargets` tool)
2. Count by status stage
3. Identify bottlenecks (targets stuck in one stage >30 days)
4. Flag unreachable targets to Managing Partner
5. Post findings as a comment on the parent sourcing issue
